Part I

Foundations

The ideas everything else stands on: what a language model is, what a neural network is, how learning works, and the handful of math operations you'll see over and over.

Chapter 1

What a language model really is

Strip away the mystique and a language model is one thing: a machine that, given some text, assigns a probability to what comes next. Everything impressive it does — answering questions, writing code, reasoning through a problem — emerges from doing this one job extraordinarily well.

Let's make "probability of what comes next" precise, because the entire field is built on it. A language model assigns a probability to a whole sequence of words, and it does so by breaking the sequence into a chain of next-token predictions using a rule from probability theory — the chain rule: the probability of a sentence is the product of the probability of each word given all the words before it.

the chain rule — the mathematical definition of a language modelP(w₁, w₂, …, wₙ) = P(w₁) · P(w₂ | w₁) · P(w₃ | w₁,w₂) · … · P(wₙ | w₁…wₙ₋₁)

The takeaway: the only quantity a language model ever needs to compute is one conditional probability — P(next token | everything so far). Master that single thing and you can both score any text (multiply the conditionals) and generate any text (sample from them). Everything in this book is either how to compute that conditional well, or what to build on top of it.

Worked example — scoring a sentence

Say a model gives these conditionals for "the cat sat":
P("the") = 0.05,  P("cat" | "the") = 0.02,  P("sat" | "the cat") = 0.10

Then P("the cat sat") = 0.05 × 0.02 × 0.10 = 0.0001.

A fluent, likely sentence earns a higher product than jumbled nonsense like "sat the cat." That number is "how likely the model thinks this text is." Training pushes real text toward high probability and gibberish toward low.

Intuition — it's a very good autocomplete

Your phone's keyboard suggests the next word from simple two-word statistics. A large language model is that idea taken to an extreme: instead of counting word pairs, it uses billions of parameters to model deep patterns — grammar, facts, reasoning, style, code. The interface is identical ("what word comes next?"), but the capability is a different universe. Every chat, essay, and line of code it writes is this autocomplete running forward, one token at a time.

Generation: the loop that produces text

Generation is just that conditional, applied repeatedly. Give the model a prompt; it outputs a probability distribution over the whole vocabulary for the next token; you pick one (the most likely, or a sample — Chapter 28); you append it; you feed the whole thing back in for the next token. Round and round until it emits a "stop" signal.

prompt → "The capital of France is" step 1 → P(next) = { "Paris":0.91, "a":0.03, "the":0.02, … } → pick "Paris" step 2 → feed "…France is Paris" → pick "." step 3 → feed "…Paris." → model emits [end] → stop result → "The capital of France is Paris."

That autoregressive loop — predict, append, repeat — is all generation ever is, from a one-word reply to a thousand-line program. Its strictly sequential nature (you can't produce token 50 without token 49) is also why generation takes time, a fact that shapes every optimization in Part VI.

Why does "predict the next word" produce something that looks like intelligence? Because to predict well across the whole internet, the model is forced to internalize whatever helps. To finish "water boils at ___" it must store facts. To finish "the opposite of hot is ___" it must learn meaning. To finish a proof or a function it must learn logic and syntax. None of this was programmed in — it was discovered, because it reduces prediction error. This is the deepest idea in the book, and it returns at every level: capability emerges from an objective.

Key idea

A language model computes one thing — P(next token | everything so far) — and everything follows. Scoring text is multiplying these probabilities (the chain rule); generating text is sampling from them in a loop; training is teaching the model to assign good ones. Hold this and nothing later is magic — it is all in service of, or built upon, next-token prediction.

Chapter 2

Neural networks in a nutshell

Under the hood, a language model is a neural network — a big mathematical function with billions of adjustable knobs. Understand the network in the abstract and everything specific (the transformer, attention) becomes far easier.

The atom is the neuron. It takes several numbers as input, multiplies each by a weight, adds them up along with a bias, and passes the total through a nonlinear activation function. That's the whole unit: a weighted sum, then a squash.

one neuronoutput = activation( w₁x₁ + w₂x₂ + … + wₙxₙ + b )

The weights are the "knobs" training adjusts — a modern LLM has billions of them, and learning is nothing but finding weight values that make outputs good (Chapter 3). Stack many neurons side by side into a layer, and stack many layers, and you have a deep neural network: numbers enter at the bottom, get transformed layer by layer, and an answer comes out the top.

Why the nonlinearity is non-negotiable

The activation looks like a footnote but it's what makes depth worth anything. Here's the proof, concretely: stack two linear layers with no activation — multiply by W₁, then by W₂. That's just multiplying by the single matrix (W₂·W₁) — one linear layer in disguise. Any number of linear layers collapses to one, and a single linear function can only draw straight lines and flat planes. It could never model "the sentiment is positive unless the word 'not' appears" — that needs a bend.

Intuition — ruler vs clay

A linear function is a rigid ruler: only straight cuts. The nonlinearity turns the network into clay — each layer can bend and fold the input space, and stacking many folds lets the network carve arbitrarily complex regions. That folding is where the ability to represent grammar, meaning, and reasoning comes from. Common activations: ReLU (zero out negatives — simple, fast), GELU/SiLU (smooth versions used in modern models), and sigmoid (squashes to 0–1, for probabilities).

A whole layer is one matrix multiply

Here's the bridge to Chapter 4. A layer computes many neurons at once, and "many weighted sums in parallel" is exactly matrix multiplication: pack the inputs into a vector and the weights into a matrix W, and the layer is activation(W·x + b). That's why models run on GPUs — GPUs do enormous matrix multiplications fast. The transformer we're building is, mechanically, a particular clever arrangement of these matrix multiplies and activations. No new atoms — only a new structure.

Chapter 3

How models learn: loss, gradients, backprop

A fresh network has random weights and outputs garbage. Learning is nudging those billions of weights, over and over, until the output is good. Three ideas drive every model in this book: a loss, its gradient, and an optimizer.

The loss — a single number for "how wrong"

First we need a score for how wrong the network currently is: the loss. For language models it's cross-entropy — large when the model puts low probability on the correct next token, small when it puts high probability there. It's literally the negative log of the probability the model assigned to the truth.

cross-entropy, the language-model lossloss = − log P(correct next token)
true word given 90% → loss = −log(0.90) = 0.105 (confident & right — tiny loss) true word given 50% → loss = −log(0.50) = 0.693 true word given 1% → loss = −log(0.01) = 4.605 (confident & wrong — big loss)

Averaged over trillions of tokens, this one number captures model quality. Notice the shape: being confidently wrong is punished far more than being merely unsure — that steep penalty is exactly what drives the model toward calibrated, correct probabilities.

The gradient — which way is downhill

The gradient answers, for every weight at once: if I nudge this weight up a hair, does the loss rise or fall, and how steeply? It's a vector pointing in the direction of steepest increase of the loss — so to reduce loss, we step the opposite way. Gradient descent is that loop: compute the gradient, take a small step downhill, repeat.

The step size is the learning rate, and it matters enormously — too small and training crawls; too large and it overshoots or explodes. Let's see it on a toy loss L(w) = (w − 3)² whose lowest point is obviously w = 3, with gradient 2(w − 3). Drag the learning rate and step through it:

Backpropagation & the optimizer

A real network has billions of weights — so how do we get the gradient for each? Backpropagation: an efficient application of calculus's chain rule that computes every weight's gradient in a single backward pass through the network. You don't need its internals; just know it's the algorithm that makes training billions of parameters feasible, and it's why every operation in the network must be differentiable.

The optimizer takes those gradients and updates the weights. Plain gradient descent uses one learning rate for everything. Modern LLMs use Adam (specifically AdamW), which is smarter: it keeps a running average of recent gradients (momentum, for smoother progress) and a running average of their squared sizes (giving each weight its own adaptive step). Faster and more stable — but at a cost that echoes through the whole book:

Why this detail matters later

Adam stores two extra numbers per weight (the two running averages), on top of the weight and its gradient. For a model with billions of parameters that's several full copies of the model in memory just to take one step — which is precisely why full fine-tuning is so memory-hungry, and why LoRA (Chapter 21) exists to sidestep it.

Key idea

All learning in this book is one loop: (1) run text through the model, (2) measure the loss, (3) backpropagate to get every weight's gradient, (4) let the optimizer step the weights downhill. Pretraining, fine-tuning, and even RLHF differ only in what text and what loss — the machinery underneath never changes.

Chapter 4

The essential math

You need surprisingly little math to understand transformers deeply — mainly three operations. Meet them concretely here, play with two of them, and the rest of the book reads smoothly.

Vectors, matrices, and the multiply

A vector is a list of numbers — a point in space. In LLMs every token becomes a vector (its embedding), and its length (say 4096 numbers) is the model's "width." A matrix is a grid of numbers; the model's learned weights live in matrices. The core operation is matrix multiplication, which transforms vectors. When you see W·x, read "reshape the vector x using the learned weights W." Worked by hand:

weight matrix W (2×3) vector x (3) W = [ 1 0 2 ] x = [ 3 ] [ 0 1 1 ] [ 1 ] [ 2 ] output row 1 = row₁ · x = 1·3 + 0·1 + 2·2 = 7 output row 2 = row₂ · x = 0·3 + 1·1 + 1·2 = 3 W·x = [7, 3] — a 3-D vector transformed into a 2-D one

"Each output = a row of weights dotted with the input" is the pattern behind every layer in the network. And notice the building block hiding inside: the dot product.

The dot product — measuring similarity

The dot product multiplies two vectors element-wise and sums the result, giving a single number. Its meaning is the key to everything ahead: it measures how aligned two vectors are. Large positive → they point the same way (similar); zero → perpendicular (unrelated); negative → opposite. Drag the two vectors and watch the number track their alignment:

Softmax — turning scores into probabilities

Finally, softmax converts a list of raw scores (any numbers, called logits) into a probability distribution — all positive, summing to 1 — while preserving order (bigger score → bigger probability). It exponentiates each score and divides by the total.

Part II

Representing Language

Before a model can process text, text must become numbers. Three steps — splitting into tokens, mapping tokens to vectors, and adding position — turn a sentence into something a transformer can compute on.

Chapter 5

Tokenization

A model never sees letters or words directly — it sees tokens: integer IDs from a fixed vocabulary. How text gets chopped into tokens shapes vocabulary size, cost, how rare words are handled, and how the model copes with other languages.

Why not just use whole words, or single characters? Both extremes fail, and seeing exactly how motivates the real solution:

  • One token per word. English alone has hundreds of thousands of words, plus names, typos, and new coinages — an unmanageably large vocabulary that still chokes on any word it hasn't seen ("out of vocabulary"). Every misspelling is a total unknown.
  • One token per character. Now the vocabulary is tiny (~100 symbols) and nothing is unknown — but sequences become ~4–5× longer (every letter is a step), which is far more expensive, and the model has to relearn spelling from scratch on every word.

The winning compromise is sub-word tokenization: frequent words become a single token, while rare words split into meaningful pieces. Common enough to be efficient, flexible enough to never be stumped.

Byte-Pair Encoding (BPE), step by step

The dominant method is BPE, and its training algorithm is simple enough to run by hand:

  1. Start with every word split into individual characters (plus an end-of-word marker).
  2. Count all adjacent token pairs across the corpus.
  3. Merge the most frequent pair everywhere into a single new token, and add it to the vocabulary.
  4. Repeat steps 2–3 until you hit a target vocabulary size (typically 50,000–150,000 tokens).

The magic is that frequency does the work: letter combinations that appear constantly (like "th", "ing", "est") get merged early and become their own tokens, while rare words stay broken into pieces. Run it yourself on a tiny corpus:

Example — a real sentence tokenized

"The cat sat" → ["The", " cat", " sat"] → IDs [791, 8415, 7731]

Notice the leading spaces are part of the tokens (" cat", not "cat") — that's how the model tracks word boundaries. And nothing is ever truly out-of-vocabulary, because worst case a word falls all the way back to individual bytes.

Why tokenization matters more than you'd think

Token count is cost and context usage — you pay per token, and the context window is measured in tokens, not words. Tokenization also explains real model quirks: models struggle with character-level tasks (counting the r's in "strawberry", reversing a string) because they see chunks, not letters. And languages whose scripts tokenize into many pieces are effectively more expensive and get less usable context per sentence than English. A humble preprocessing step with outsized downstream consequences.

Chapter 6

Embeddings

A token ID like "8415" is just a label — it carries no meaning a network can compute with. The embedding turns each token into a vector of numbers, positioned so that meaning becomes geometry.

The embedding matrix is a big lookup table: one row per vocabulary token, each row a vector of length d_model (the model's width, e.g. 4096). Tokenizing gives you IDs; you grab the matching rows; and you have a sequence of vectors — the actual input to the transformer. Here's the lookup, concretely, with a toy 4-dimensional table:

embedding table (toy, d=4) ID 8415 "cat" → [ 0.2, −0.4, 0.7, 0.1 ] ID 7731 "sat" → [−0.3, 0.6, 0.2, −0.5 ] ID 791 "The" → [ 0.9, 0.1, −0.2, 0.4 ] "The cat sat" → IDs [791, 8415, 7731] → look up rows → [[0.9,0.1,−0.2,0.4], [0.2,−0.4,0.7,0.1], [−0.3,0.6,0.2,−0.5]]

Crucially, these vectors are learned parameters — they start as random noise and training shapes them. And what training shapes them into is the beautiful part.

After training, embeddings arrange themselves so that similar words sit near each other and relationships become directions. The famous demonstration: the vector arithmetic king − man + woman lands right next to queen. The "royalty" difference (man→king) is the same direction as (woman→queen), so you can do analogies with vector addition. Static vs contextual — what the transformer adds

A bit of history clarifies what's new. Early methods like word2vec and GloVe gave each word one fixed vector regardless of context — so "bank" (river) and "bank" (money) were forced to share a single embedding, an obvious problem. The transformer's breakthrough is contextual embeddings: the input embedding is only a starting point, and each layer of attention (Part III) refines a word's vector based on its neighbors.

Example — "bank" in context

"river bank" → after attention, bank's vector shifts toward {water, shore, geography}.
"savings bank" → the same starting vector shifts toward {money, account, finance}.

Same input embedding, two different contextual representations — because attention mixed in the neighbors. This is why LLMs handle ambiguity that static embeddings couldn't.

Key idea

Embeddings map discrete tokens into a continuous space where meaning is geometry and relationships are directions. They're learned, not hand-built. The input embedding is a fixed starting vector per token; the entire job of the transformer layers above is to progressively enrich each vector with context, turning static meaning into contextual meaning.

Chapter 7

Positional information

Attention, as we'll see, is order-blind — on its own it treats a sentence as an unordered bag of words. "Dog bites man" and "man bites dog" would produce identical representations. So before feeding tokens to the transformer, we must inject where each one sits.

Why attention loses order

Here's the problem, precisely. Attention computes each token's new representation as a weighted average over all tokens (Chapter 9). Averaging is a set operation — it doesn't care about order. Shuffle the inputs and the same set of tokens produces the same averages. So "dog bites man" and "man bites dog" contain the identical set {dog, bites, man} and would be indistinguishable to raw attention — even though they mean opposite things. The fix is to make each token's vector encode its position, so position 1 and position 3 are genuinely different inputs.

Sinusoidal positional encoding

The original transformer added a positional encoding to each token's embedding: a vector, unique per position, built from sine and cosine waves at many different frequencies. Because each dimension oscillates at a different rate, every position gets a distinct "fingerprint."

sinusoidal positional encodingPE(pos, 2i) = sin(pos / 10000^{2i/d})    PE(pos, 2i+1) = cos(pos / 10000^{2i/d})
position 1, dim 0 → sin(1 / 10000⁰) = sin(1) = 0.84 position 1, dim 1 → cos(1) = 0.54 position 2, dim 0 → sin(2) = 0.91 early dims → high frequency (change fast across positions) later dims → low frequency (change slowly) — together they uniquely tag every position

What modern models use: RoPE

Most models since ~2022 replaced additive sinusoids with Rotary Position Embeddings (RoPE). Instead of adding a position vector, RoPE rotates each token's query and key vectors by an angle proportional to its position (position 1 rotates a little, position 100 a lot). The elegant consequence falls out of the dot product: when a query at position m is compared with a key at position n, the rotations combine so that only the difference (m − n) matters.

query rotated by m·θ, key rotated by n·θ their dot product depends on → (m·θ − n·θ) = (m − n)·θ ⇒ attention sees RELATIVE distance, not absolute position

This is why RoPE is the modern standard: relative position is what usually matters ("the adjective two words back"), and because the rotation is a smooth function, RoPE extrapolates to sequences longer than those seen in training far better than fixed sinusoids — the basis for the long-context techniques in Chapter 31.

Key idea

Attention has no built-in sense of order, because averaging is a set operation — so position must be injected. Older models add sinusoidal fingerprints to the embeddings; modern models rotate the query/key vectors (RoPE) so that relative distances fall out of the dot product naturally and extend to long contexts. Either way, the goal is the same: give the model a way to know where each token is.

Part III

The Transformer

The architecture behind every modern LLM. We build it from its heart — attention — outward to the full stack, then to the modern refinements that power today's models.

Chapter 8

Attention: the core idea

Attention is the single most important idea in modern AI, and it's simpler than its reputation. It lets each token look at every other token and pull in the information it needs — turning a bag of isolated word-vectors into a web of context.

The cleanest way to understand it: attention is a soft dictionary lookup. A normal dictionary (or a hash map) takes an exact key and returns its one value. Attention is the fuzzy, blended version: every token poses a question, every token advertises what it offers, and you retrieve a weighted mixture of everyone's contents — weighted by how well each match your question.

Intuition — query, key, value (the library analogy)

Imagine searching a library. Your Query is what you're looking for ("books about volcanoes"). Each book has a Key — its spine label advertising its topic. You compare your query against every key to see how well they match. And each book has a Value — its actual contents, which you take away in proportion to how well its key matched. Attention does exactly this: each token emits a Query ("what am I looking for?"), a Key ("what do I offer?"), and a Value ("what I'll contribute if attended to"), and each token's new representation is a blend of the Values, weighted by Query·Key match.

Where do Q, K, and V come from? Each is produced by multiplying a token's embedding by a learned weight matrix — W_Q, W_K, W_V. Three different projections, three different "views" of the same token. Training shapes these matrices so that useful things match: pronouns' queries come to match their referents' keys, adjectives match their nouns, verbs match their subjects. The mechanism is fixed and simple; the intelligence is in what the learned projections decide to make "similar."

Example — resolving "it"

"The trophy didn't fit in the suitcase because it was too big."

To understand "it," the model needs to know it refers to the trophy. Attention does this: "it" emits a query that (after training) matches the trophy's key more than the suitcase's, so it pulls the trophy's value into its own representation. The word "it" literally becomes "trophy-flavored." Change "big" to "small" and a well-trained model shifts its attention to "suitcase" instead.

Chapter 9

Self-attention mechanics

Now the actual computation, which is just the three operations from Chapter 4 in sequence. It's called self-attention because Q, K, and V all come from the same sequence — the tokens attend to one another.

Given the input vectors X, the whole thing is one formula:

scaled dot-product attentionAttention(Q,K,V) = softmax( Q Kᵀ / √d_k ) V

Five steps, each one an operation you already know:

  1. Project — compute Q = X·W_Q, K = X·W_K, V = X·W_V. Three views of every token.
  2. Score — Q·Kᵀ is the dot product of every query with every key: a grid of scores, "how much does each token want each other token?"
  3. Scale — divide by √d_k. Dot products grow with dimension; without this, scores get large and softmax saturates into a near-hard pick. Scaling keeps gradients healthy.
  4. Softmax — turn each row of scores into weights summing to 1: a probability distribution over "who to attend to."
  5. Blend — multiply the weights by V. Each token's output is a weighted average of all value vectors; tokens it cared about dominate.

Chapter 10

Multi-head attention

One attention operation captures one kind of relationship at a time. Language needs many at once — subject–verb, noun–adjective, pronoun–referent, nearby-vs-distant. The fix: run several attention operations in parallel, each in its own subspace.

With h heads, we don't run full-size attention h times (that would be h× the compute). Instead we slice the model dimension into h smaller pieces. Each head gets its own small Q/K/V projections into a d_k = d_model / h–dimensional subspace, computes attention there, and produces a small output. The heads run independently — then their outputs are concatenated back to full width and passed through a final projection W_O that lets the heads' findings mix.

multi-head attentionheadᵢ = Attention(X·W_Qⁱ, X·W_Kⁱ, X·W_Vⁱ)  →  MultiHead = Concat(head₁…headₕ)·W_O
typical config: d_model = 512, heads h = 8 per-head dimension d_k = 512 / 8 = 64 each head does attention in its own 64-dim slice (8 different "perspectives") concatenate the 8 outputs: 8 × 64 = 512 → back to full width, then × W_O
Intuition — many specialists, same cost

Instead of one generalist looking at everything, you have (say) 8 or 32 specialists, each examining the sequence through a different lens in a smaller subspace. Because you split the width rather than adding to it, total compute is about the same as a single full-width head — but the model gains the ability to track multiple relationship types simultaneously. It's parallelism of perspective. Interpretability research has even found specific heads with clear jobs — e.g. "induction heads" that copy patterns, and heads that track the previous token or syntactic dependencies.

Chapter 11

Completing the block

Attention mixes information across tokens. The rest of a transformer block lets the model compute on each token and keeps deep training stable. Three pieces finish it — then we stack the block N times and we're done.

Residual connections

Each sub-layer's output is added to its input: x + SubLayer(x). This "residual" (or skip) connection means each layer computes a correction to the running representation rather than replacing it. It's what lets networks be dozens of layers deep: gradients flow straight down the addition path during backprop, sidestepping the "vanishing gradient" problem that used to make deep nets untrainable. The mental model is a residual stream — one vector per token flowing straight up the stack, which each layer reads from and writes a small correction back into.

Normalization

Layer normalization re-centers and re-scales each token's vector to keep activations from drifting or exploding as corrections accumulate across layers. Concretely, it subtracts the mean and divides by the standard deviation, then applies a learned scale and shift:

LayerNorm on x = [2, 4, 6, 8] mean μ = (2+4+6+8)/4 = 5 variance = [(2−5)²+(4−5)²+(6−5)²+(8−5)²]/4 = (9+1+1+9)/4 = 5, std ≈ 2.24 normalize (x−μ)/std → [−1.34, −0.45, 0.45, 1.34] (now mean 0, std 1)

Modern models use RMSNorm, a cheaper variant that skips the mean-centering (just divides by the root-mean-square), and apply normalization before each sub-layer ("pre-norm") rather than after — which trains far more reliably at depth.

The feed-forward network (MLP)

After attention, each token passes independently through a small two-layer network: expand to a wider hidden size (typically 4×), apply a nonlinearity, compress back. This feed-forward network is where the model does per-token "thinking," and where most of its parameters — and much of its stored factual knowledge — actually live.

feed-forward networkFFN(x) = W_down · activation( W_up · x )   [512 → 2048 → 512]

The original used ReLU; modern models use SwiGLU, a gated activation that improves quality per parameter. Note the split of labor: attention moves information between tokens; the FFN processes each token's information. A block needs both.

Chapter 12

Encoder, decoder, and cross-attention

The 2017 paper introduced two stacks — an encoder and a decoder — because it was built for translation. Understanding both clarifies why today's LLMs keep only one.

The encoder reads the entire input at once, bidirectionally — every token attends to every other, past and future, no restrictions. It builds a deep representation of the whole input, ideal for understanding tasks. The decoder generates output one token at a time, and it must be causal: each position may only attend to earlier positions, never the future — because at generation time the future tokens don't exist yet.

The causal mask

That restriction is enforced by a causal mask: before the softmax, the scores for all future positions are set to −∞, so after softmax their weights are exactly 0. It's a small change with a huge consequence — it's what makes a model generative (able to produce text left-to-right) rather than just an analyzer. You saw it directly in the attention simulator (Chapter 9): toggle the mask on and the upper triangle of the grid goes dark — "sat" can no longer peek at "quietly."

raw scores for a query, then causal masking (it sits at position 2): [ the, cat, sat, quietly ] = [0.00, 0.58, 0.00, 0.58] mask the future (position 3) → [0.00, 0.58, 0.00, −∞] softmax → [0.26, 0.47, 0.26, 0.00] the future gets zero weight

Cross-attention: the bridge

In the encoder–decoder design, the decoder must consult the input. It does so with cross-attention: identical math to self-attention, but the queries come from the decoder (what it's generating) while the keys and values come from the encoder's output (the processed input). So while producing a translation, each output word can reach back to the relevant source words.

Intuition — self vs cross attention

In self-attention, a sequence attends to itself (Q, K, V all from one place). In cross-attention, one sequence attends to another (Q from here, K/V from there). Translating "the cat" → French: as the decoder writes "chat," its query lights up the encoder's key for "cat" and pulls that meaning across the bridge. Same mechanism, different sources.

Because the causal decoder is what modern LLMs keep, the next chapters build entirely on it.

Chapter 13

The three families & why decoder-only won

Three architectures grew from the transformer, each making a different bet. Modern LLMs almost universally chose one — and knowing why is knowing what an LLM fundamentally is.

Why did the field collapse onto decoder-only? Not because the others are wrong — encoder-only models are still excellent for search and classification — but because decoder-only is simpler and scales better, and at scale that beats built-in cleverness. Four compounding reasons:

  • The objective is free. "Predict the next token" needs no human labels — every sentence ever written is training data. This soaks up the entire internet in a way the other families' supervised objectives can't match.
  • One stack, no bridge. No separate encoder, no cross-attention to wire up, no architectural asymmetry. Fewer moving parts means cleaner scaling and simpler engineering — which matters enormously at billion-parameter, thousand-GPU scale.
  • Everything becomes text-to-text. Translation, Q&A, classification, code — frame the task in the prompt and the model just continues it. The prompt is a prefix to condition on, which quietly absorbs the job the encoder used to do.
  • In-context learning emerges. Because the prefix can be anything, a large causal model learns to use examples given right in the prompt at inference time (few-shot prompting) — with no weight updates at all. This remarkable capability (Chapter 35) is hard to get any other way, and it fell out "for free" from scaling decoder-only models.
Key idea

A modern LLM is a decoder-only transformer: a single causal stack trained to predict the next token. The causal mask makes it generative; the simplicity makes it scalable; the "everything is text" framing makes it universal. When we say "the model" for the rest of this book, we mean this.

Chapter 14

Modern architecture refinements

The 2017 transformer works, but every frontier model changes five things about it. These aren't trivia — each fixes a concrete problem with the original, and understanding how each works is what separates "read the paper" from "knows how models are actually built." We'll take them one at a time, mechanism first.

1. RMSNorm + pre-norm — cheaper, more stable normalization

LayerNorm (Chapter 11) does two things to each vector: subtract its mean (re-center to 0), then divide by its standard deviation (re-scale to unit spread). RMSNorm asks: is the mean-subtraction actually necessary? Empirically, no — so it drops it and only rescales, dividing by the root-mean-square (the square root of the average of the squared entries):

LayerNorm: (x − mean) / std · γ + β ← 2 statistics (mean & variance), a shift β RMSNorm: x / √(mean(x²) + ε) · γ ← 1 statistic (RMS), no subtraction, no shift worked on x = [2, 4, 6, 8]: mean(x²) = (4 + 16 + 36 + 64) / 4 = 30 RMS = √30 ≈ 5.48 x / RMS = [0.37, 0.73, 1.10, 1.46]

One fewer statistic and no subtraction makes it noticeably faster, and models train just as stably with it. The second half, pre-norm, is about where you normalize. The original put the norm after each sublayer (attention → add → norm); modern models put it before (norm → attention → add). Why it matters: pre-norm leaves the residual stream (Chapter 11) as an untouched highway from top to bottom, so gradients flow cleanly to the earliest layers. That's what lets an 80-layer model train without the loss exploding — post-norm deep stacks tend to destabilize.

2. SwiGLU — a smarter feed-forward, via gating

The original FFN is W_down · ReLU(W_up · x): expand, apply a fixed nonlinearity, compress. SwiGLU replaces the single activation with a gate. Instead of one projection, it makes two projections of the input — a "value" path and a "gate" path — and multiplies them element-wise, with the gate passed through a smooth activation (SiLU, a.k.a. swish):

plain FFN: W_down · ReLU( W_up · x ) SwiGLU: W_down · ( SiLU(W_gate·x) ⊙ (W_up·x) ) └── gate ──┘ └─ value ─┘ (⊙ = element-wise ×) what a gate does, per hidden unit: gate 0.9, value 2.0 → 1.8 passes through gate 0.1, value 2.0 → 0.2 passes

The gate is a learned, input-dependent volume knob on every hidden neuron: for this particular token, how much of each value should flow through? A plain ReLU can only switch a unit fully on or off; a multiplicative gate can pass 30% of one unit and 90% of another, smoothly. That extra expressiveness is why gated FFNs beat ReLU/GELU at equal size. The cost is a third weight matrix (gate, value, down instead of up, down), so implementations shrink the hidden width to ~⅔ to keep the total parameter count matched.

3. Grouped-Query Attention (GQA) — shrinking the KV cache

This one only makes sense once you know the KV cache (Chapter 29), so here's the seed: during generation, the model stores the Key and Value vectors of every past token so it doesn't recompute them. With standard multi-head attention (Chapter 10), that means storing K and V for every head — and for a long context that memory is enormous. GQA's observation: queries are recomputed cheaply each step, but K/V are what's stored, so share the K/V heads across groups of query heads while keeping all the query heads:

MHA: 32 query heads, 32 K/V heads → KV cache stores 32 sets GQA: 32 query heads, 8 K/V heads → 4 queries share each K/V → cache 4× smaller MQA: 32 query heads, 1 K/V head → all share one → 32× smaller (a little quality loss) concrete, per layer (dim 128/head, 4096-token context, bf16): MHA cache = 2 × 32 × 128 × 4096 × 2 B ≈ 67 MB/layer GQA (8 K/V) ≈ 17 MB/layer — same 4× cut, negligible quality drop

Across dozens of layers and many concurrent users, that 4× is the difference between serving a handful of long conversations and serving hundreds. It's standard in Llama, Mistral, and most current models.

4. Mixture of Experts (MoE) — decoupling size from cost

Instead of one big FFN per layer, an MoE layer holds many smaller "expert" FFNs plus a tiny router. For each token, the router scores all experts, picks just the top few (often 2), runs only those, and combines their outputs weighted by the router's scores:

router (a small linear layer): g = softmax(W_r · x) → e.g. experts scored [0.05, 0.40, 0.02, 0.35, 0.08, …] pick top-2 → expert 2 and expert 4; renormalize their weights → 0.53, 0.47 output = 0.53 · Expert₂(x) + 0.47 · Expert₄(x) (the other experts stay idle)

This decouples capacity from compute — you get the knowledge of a giant model at the running cost of a small one. The catch is memory (you must store every expert) and a training subtlety: without a load-balancing loss that nudges the router to spread tokens evenly, it collapses to always using the same one or two experts, wasting the rest.

Intuition — a panel of specialists

A dense model is one generalist reading every token. An MoE is a large panel of specialists with a receptionist (the router) who forwards each token to the two or three most relevant specialists. You get the breadth of a huge model at the cost of a small one — you just have to keep all the specialists on payroll (in memory). Mixtral and several frontier systems are MoE.

5. FlashAttention — same math, hardware-aware

This one changes nothing about the attention formula — it changes how it's computed on the GPU, and it's why long contexts became affordable. Standard attention builds the full score matrix S = QKᵀ, stores it, softmaxes it, then multiplies by V. For a 8,192-token sequence that matrix is ~67 million numbers per head — far too big for the GPU's fast on-chip memory (SRAM), so it gets shuttled back and forth to slow memory (HBM), and that shuttling, not the arithmetic, is the bottleneck.

FlashAttention never builds the whole matrix. It walks through K/V in tiles small enough to fit in fast SRAM, and computes the softmax incrementally using the "online softmax" trick — keeping a running maximum and running sum so it can normalize correctly without ever having seen all the scores at once — accumulating the output tile by tile. The result is bit-for-bit the same as standard attention, but memory drops from growing with the square of the length to growing only linearly, and it avoids the slow-memory round-trips, so it's several times faster too. Because it's a systems optimization rather than a math change, the model's outputs are identical — you just get them cheaper.

Key idea

Today's models = the 2017 transformer + RoPE (relative position, Chapter 7) + RMSNorm/pre-norm (cheap, stable) + SwiGLU (gated FFN) + GQA (small KV cache), often with MoE (capacity without compute) and FlashAttention (same math, faster). The skeleton is unchanged; each refinement fixes one concrete problem — position, stability, expressiveness, memory, scale, speed. Knowing the mechanism of each is the difference between the textbook and the frontier.

Part IV

Pretraining

How a base model is born: the objective that teaches it the world, the data that feeds it, the laws that govern how big to build it, and the engineering that makes training at scale possible.

Chapter 15

The pretraining objective

Pretraining is where the model learns almost everything it knows — and the objective is exactly the next-token prediction from Chapter 1, run on a staggering amount of text.

Take trillions of tokens of raw text (web pages, books, code) and, at every position, ask the model to predict the next token, scoring it with cross-entropy loss (Chapter 3). A crucial efficiency makes this feasible: because the decoder is causal (each position sees only its past), a single forward pass over a sequence produces a prediction at every position at once — one sentence yields many training signals in parallel. This is teacher forcing.

Example — one sequence, many lessons

• after "The cat sat" → "quietly"

Every position is a mini prediction problem, all graded simultaneously. Multiply by trillions of tokens and the model is forced to learn grammar, facts, reasoning, and code — because all of them reduce loss.

The sheer scale is what makes pretraining both powerful and expensive. A useful rule of thumb ties the pieces together:

training compute (FLOPs) ≈ 6 · N · D (N = parameters, D = training tokens) e.g. an 8B model on 15T tokens → 6 × 8e9 × 15e12 ≈ 7 × 10²³ FLOPs = months of training across thousands of GPUs this one stage is ≈ 99% of a model's total lifetime compute

The output of pretraining is a base model: a brilliant, knowledgeable text-continuer that does not yet know it should follow instructions or be helpful. Ask it a question and it might continue with three more questions, because that's a plausible continuation of the text. Everything in Part V is comparatively cheap polish on top of this expensive foundation — which is why the factor 6ND matters so much, and why the next chapter (scaling laws) is really about spending it wisely.

Chapter 16

Data at scale

A model is what it eats. The quality, quantity, and mix of pretraining data shape the model more than almost any architectural choice — and preparing it is a huge, underappreciated engineering effort.

Raw web text is mostly unusable — spam, boilerplate, duplicates, toxic content. Turning it into good training data means a heavy pipeline, run over the entire corpus:

  • Filtering — remove low-quality pages using heuristics and trained quality classifiers; keep text resembling the useful parts of the web (well-written, informative).
  • Deduplication — remove repeated and near-duplicate documents. Duplicates waste compute, cause memorization, and can raise held-out loss. At scale this uses hashing and MinHash/LSH for near-duplicates (the same technique family from the vector-search world).
  • Mixing — balance the sources, which strongly shapes ability.
  • Decontamination — remove any text overlapping evaluation benchmarks, so test scores aren't inflated by memorization (revisited in Chapter 34).

Deduplication is worth a closer look because "near-duplicate" is subtle — the same article reworded, or a page repeated with a different header, isn't caught by exact hashing. The trick (MinHash + LSH) makes fuzzy matching cheap at web scale: chop each document into overlapping word-chunks ("shingles"), hash them, and keep just the few smallest hash values as a compact "signature." Two documents that share most of their shingles will, by construction, share most of their signature — so comparing signatures approximates comparing full documents, at a tiny fraction of the cost. LSH then groups documents whose signatures collide into the same bucket, so you only compare likely matches instead of every pair. Remove the collisions and the near-duplicates are gone.

an illustrative pretraining mix (proportions vary by lab): filtered web ~67% code ~10% books ~10% wikipedia ~3% math ~5% multilingual ~5% more code & math in the mix → measurably better reasoning, even on non-code tasks
Key idea

"Garbage in, garbage out" is doubly true at scale. Two models with identical architecture and size can differ enormously based on data quality alone. The frontier is increasingly about data — what to include, how to clean it, how to mix it — as much as bigger networks. And how much data you need is set by the scaling laws next.

Chapter 17

Scaling laws

One of the most consequential discoveries in AI: model performance improves predictably as you scale up compute, data, and parameters. This turned model-building from guesswork into planning.

Scaling laws show that the loss falls smoothly as a power law of model size, dataset size, and compute — so you can predict a large model's loss from small experiments and decide how big to build before spending millions. The landmark refinement, the Chinchilla result, answered a budgeting question: given a fixed compute budget C ≈ 6·N·D, how should you split it between parameters N and training tokens D?

two real runs at similar compute (~3–6 × 10²³ FLOPs): GPT-3: 175B params, 300B tokens → 1.7 tokens/param (too big, undertrained) Chinchilla: 70B params, 1.4T tokens → 20 tokens/param (compute-optimal) smaller-but-better-fed Chinchilla beat the larger GPT-3.

The surprising lesson: earlier models were too big and undertrained. For compute-optimal training you scale parameters and tokens together (Chinchilla's rule of thumb: ~20 tokens per parameter). Chapter 18

Training at scale

A frontier model has far more parameters than fit on any single GPU, and trains for months across thousands of them. To see why the techniques below exist, first look at the memory that makes single-GPU training impossible.

a 70B-parameter model in training (recall Adam's overhead, Chapter 3): weights (bf16) 70B × 2 = 140 GB gradients (bf16) 70B × 2 = 140 GB Adam states (fp32, ×2) 70B × 8 = 560 GB ─────────────────────────────────────── ≈ 840 GB — and a top GPU holds ~80 GB, about 1/10th of it

So the model must be split across many GPUs — and the whole art is splitting it while keeping them all busy and talking efficiently. There are three ways to split, and they're combined.

Data parallelism — split the batch

The simplest. Give every GPU a full copy of the model but a different slice of the training batch. Each computes gradients on its slice independently; then all GPUs average their gradients together (a communication step called an "all-reduce") so every copy applies the same update and stays identical. This multiplies throughput — 1,000 GPUs process 1,000 batches' worth of data at once. What it does not solve: memory, because every GPU still holds the entire model. Data parallelism alone works only when the model already fits on one GPU.

Tensor parallelism — split each matrix

When a single layer is too big, split the matrices themselves. A 4096×4096 weight matrix can be sliced column-wise into four 4096×1024 pieces, one per GPU. Each GPU multiplies the input by its slice, producing part of the output; the partial results are then combined across GPUs to reconstruct the full output vector. Now no single GPU ever holds a whole matrix. The cost is communication on every layer (the GPUs must exchange partial results constantly), so tensor parallelism is used only within a tightly-connected group of GPUs (typically the 8 in one server, linked by fast interconnect).

Pipeline parallelism — split the layers

Put different layers on different GPUs: GPU 0 holds layers 1–20, GPU 1 holds 21–40, and so on. A batch flows through like an assembly line. The obvious problem: while GPU 1 waits for GPU 0 to finish, it sits idle — the "pipeline bubble." The fix is to chop the batch into many micro-batches and feed them in a staggered stream, so that once the pipeline fills, every GPU is always working on some micro-batch. It's the same trick a factory uses to keep every station on the line busy.

Sharding (ZeRO / FSDP) — stop duplicating what you don't need

Data parallelism wastes memory by storing the full weights, gradients, and optimizer states on every GPU. ZeRO (and its PyTorch form, FSDP) fixes this by sharding those across the GPUs — each GPU permanently holds only its 1/N slice of the optimizer states, gradients, and even the weights. When a particular layer is needed for computation, its weight shards are briefly gathered from all GPUs to reconstruct the full layer, used, then immediately released. This trades a little extra communication for an enormous memory saving, and it's the key technique that lets an 840 GB model spread across, say, 16 GPUs with ~50 GB each.

Two memory-savers that stack on top

Mixed precision. Store and compute in 16-bit numbers instead of 32-bit: half the memory, and modern GPUs run 16-bit matrix multiplies ~2× faster. The subtlety is which 16-bit format — bf16 keeps the full numeric range of 32-bit (just with less precision), so gradients never silently overflow or underflow to zero; the older fp16 has a narrow range and needs a "loss scaling" workaround. A master copy of the weights is still kept in fp32 for accuracy.

Gradient checkpointing. During the forward pass, the intermediate activations of every layer are normally kept in memory because the backward pass needs them to compute gradients — and there are a lot of them. Checkpointing keeps only a handful of "checkpoint" activations and recomputes the rest on the fly during the backward pass. It trades roughly 30% more compute for a large drop in activation memory — often the difference between a batch fitting or not.

Key idea

Training at scale is a systems problem: an 840 GB model can't fit on an 80 GB chip, so you split it. Data parallelism splits the batch (throughput), tensor parallelism splits each matrix (fits big layers), pipeline parallelism splits the layers (fits deep models), and FSDP/ZeRO shards the optimizer states so nothing is needlessly duplicated — with mixed precision and gradient checkpointing squeezing memory further. Real runs combine all of these. You don't need to implement it, but now you know what each word actually does.

Turning a knowledgeable text-continuer into a helpful, aligned assistant. This is the richest part of the modern pipeline — teaching behavior, learning from preferences, and reinforcing correctness.

Chapter 19

Intuition — the onboarding of a genius

Think of the base model as a brilliant new hire who's read everything but never had a conversation. SFT is shadowing a senior colleague to learn the job's format. Reward modeling is training a manager who can grade work. RL is the employee practicing, getting graded, and improving. Each stage teaches something the previous couldn't — behavior you can demonstrate, then quality you can only compare, then correctness you can verify.

A key axis throughout: online methods (the model generates its own answers during training and learns from them — powerful, expensive) versus offline methods (train on a fixed dataset of good/bad answers — simpler, cheaper). And a special case: when correctness is checkable (math, code), you can replace the learned grader with a program — the un-foolable path.

Every method below comes with a calculator in this chapter so you can drive the math yourself — the reward-model loss, the PPO clip, DPO, and GRPO advantages.

Chapter 20

Supervised fine-tuning (SFT)

The simplest and most impactful post-training stage. Show the model thousands of examples of an assistant behaving well, and it learns to imitate — using the very same next-token loss as pretraining (Chapter 3). Nothing new in the machinery; only the data changes.

Each example is a conversation wrapped in a chat template with role markers (system, user, assistant). The essential trick that makes it "fine-tuning an assistant" rather than "more pretraining": we tokenize the whole conversation but mask the prompt out of the loss. The model is graded only on generating the assistant's tokens — never on predicting the user's words. In code, masked positions get the label -100, which the loss function ignores.

tokens: <user> Summarize photosynthesis <assistant> Plants convert sunlight … <end> labels: [ -100 -100 -100 -100 Plants convert sunlight … <end> ] └────────── prompt: IGNORED ──────────┘ └────── graded (loss here) ──────┘ ⇒ gradient flows only from the assistant's tokens → the model learns to respond, not to mimic prompts
Intuition — why mask the prompt

You're training an assistant, not a user-simulator. If you graded the model on predicting the user's words too, you'd waste capacity learning to write questions and bias it toward your prompt style. Masking points every gradient at the one skill you care about: producing a great response given a prompt.

Example — one SFT pair

PROMPT (masked): "Summarize photosynthesis in one sentence."
RESPONSE (trained on): "Plants convert sunlight, water, and CO₂ into glucose and oxygen."

Thousands of pairs like this — coding, writing, refusals, chat — teach the behavior of a helpful assistant. It's learning the job, not new facts.

SFT runs for just 1–3 epochs (it overfits fast) with a small learning rate (~1e-5). What it teaches: instruction-following, formatting, tone, refusals, chat structure. What it can't reliably do: install substantial new knowledge — that lived in pretraining, and forcing new facts here tends to teach confident hallucination instead. The famous "LIMA" finding: ~1,000 carefully curated examples can beat tens of thousands of mediocre ones. SFT quality is dominated by data quality, not quantity — which is why the real work is the data pipeline (dedup, filtering, decontamination), not the training loop.

Chapter 21

Parameter-efficient fine-tuning (PEFT)

Full fine-tuning updates all of a model's billions of weights — and it's memory-hungry not because of the weights, but because of everything the optimizer stores around them. PEFT gets most of the benefit while training a tiny fraction of the parameters.

Recall from Chapter 3 that Adam keeps two extra numbers per weight. So full fine-tuning holds several copies of the model in memory at once:

7B model, full fine-tuning with Adam weights + gradients (bf16) ≈ 28 GB Adam optimizer states (fp32, ×2) ≈ 56 GB ← the real cost ───────────────────────────────────────── ≈ 84 GB before activations → multiple high-end GPUs for a "small" model

LoRA — the dominant method

LoRA (Low-Rank Adaptation) rests on a finding: the change fine-tuning makes to each weight matrix is "low-rank" — it carries little independent information, so it can be squeezed into two skinny matrices. LoRA freezes the original weight W and learns a small update B·A, where the shared dimension (the rank r) is tiny. You train only B and A.

LoRAW′ = W_frozen + (α/r)·B·A   B is d×r, A is r×d, r ≪ d

QLoRA and other PEFT methods

QLoRA quantizes the frozen base to 4-bit (Chapter 30) while keeping the LoRA adapters at full precision — letting you fine-tune a 65B model on a single GPU. Other approaches: prefix/prompt tuning (learn a few "virtual tokens" prepended to the input, freezing the whole model), adapters (small inserted layers), and newer LoRA refinements (DoRA, rsLoRA). LoRA/QLoRA dominate in practice.

When full fine-tuning still wins

LoRA's premise — a low-rank update — holds for adaptation (style, instructions, moderate domain shift). It can underfit when the change is genuinely large (teaching a new language, injecting lots of new knowledge). Rule of thumb: LoRA for behavior, full fine-tuning for big knowledge shifts. For instruction- and preference-tuning — the bulk of post-training — LoRA is near-parity and vastly cheaper.

Chapter 22

Learning from preferences: reward modeling

SFT teaches behavior you can demonstrate. But "be more helpful" is hard to demonstrate and easy to judge. So we collect comparisons and train a model to predict them — an automatic grader for the next stage.

The insight is an asymmetry: nobody can reliably write the perfect answer, but anyone can pick the better of two. So we collect preference data — the model produces two responses, a human (or AI) picks the better — and train a reward model to assign a scalar score where preferred answers score higher.

The Bradley-Terry model

We assume each response has a hidden quality score r. The Bradley-Terry model says the probability a human prefers the winner w over the loser l is the sigmoid of their score difference — so a bigger reward gap means a more confident preference. Training minimizes the negative log of that probability, which pushes the chosen answer's score above the rejected one's.

Reward hacking (Goodhart's law)

"When a measure becomes a target, it ceases to be a good measure." The reward model is a proxy for human preference, trained on limited comparisons. Optimize it too hard and the model finds answers that score high but humans dislike — padded verbosity, sycophancy, format tricks. True quality rises, peaks, then falls as you over-optimize the proxy (you'll see this "frontier" plotted in Chapter 33). This failure haunts all of RLHF, and it motivates the KL leash next — and the un-foolable verifiers in Chapter 25.

Chapter 23

RLHF with PPO

The classic recipe behind the original ChatGPT. Now that we have a grader (the reward model), the model practices against it using reinforcement learning. First, what RL even is.

Reinforcement learning in one paragraph

RL is learning by trial and error. An agent takes actions in an environment and receives rewards; it learns a policy (a strategy) that maximizes reward over time. Unlike supervised learning, there's no "correct answer" to copy — only a reward saying how good what you did was, so the agent must discover good behavior itself. For an LLM: the policy is the model, an action is generating a token, the environment is the text so far, and the reward comes from the reward model scoring the finished response.

The objective and the KL leash

The goal: maximize reward, but don't drift too far from the SFT model. That second clause is essential — without it, the model chases reward off a cliff into the degenerate text that hacks the reward model (Chapter 22).

the RLHF objectivemaximize  𝔼[ reward(x,y) ] − β · KL( π ‖ π_ref )

The KL penalty measures how far the policy π has drifted from a frozen reference π_ref (the SFT model); β sets the leash's strength. It keeps the model where the reward model is still trustworthy, and preserves fluency. Watch it catch a hack:

a response scores a high reward 3.5 — but drifted a lot from the reference (KL = 40) net objective = reward − β·KL = 3.5 − 0.1·40 = −0.5 ← net NEGATIVE the leash cancelled a "high-reward" answer that was really reward hacking. a modest, on-distribution answer (KL=10): 3.5 − 0.1·10 = +2.5 ← rewarded

PPO: taking careful steps

PPO (Proximal Policy Optimization) is the algorithm that does the update. Its worry: one over-large step can destabilize training. So it tracks the probability ratio between the new and old policy for each token, multiplies it by the token's advantage (how much better than expected the outcome was), and clips the ratio to a small trust region (typically ±20%), taking the pessimistic of clipped and unclipped. This caps how far one update can move the policy.

Chapter 24

DPO & direct alignment

DPO's insight is almost a magic trick: the elaborate RLHF objective has a known mathematical solution, and if you rearrange the algebra, the reward model cancels out. You can train directly on preference pairs with a simple loss — no reward model, no RL loop, no four-model tax.

The paper's title captures it: "Your Language Model Is Secretly a Reward Model." By expressing the optimal reward in terms of the policy itself and substituting into the Bradley-Terry formula (Chapter 22), the intractable parts cancel, leaving a clean loss over (chosen, rejected) pairs:

DPO lossL = − log σ( β · [ log(π(y_w)/π_ref(y_w)) − log(π(y_l)/π_ref(y_l)) ] )

Read it as the reward-model loss with the reward replaced by an implicit reward β·log(π/π_ref) — how much more likely the current policy makes a response than the frozen reference does. DPO simply pushes the chosen response's implicit reward above the rejected one's. Watch the loss respond to the margin:

The variant family

DPO has quirks (it can over-fit clear preferences and lengthen responses), and variants patch each: IPO (regularizes against over-fitting), KTO (works on unpaired thumbs-up/down data), ORPO (fuses SFT + preference in one stage, no reference model), SimPO (reference-free, length-normalized). Iterative/online DPO — generate fresh responses, label them, run DPO, repeat — recovers much of PPO's benefit at a fraction of the complexity, and is a common modern default.

Key idea

DPO collapses reward modeling and RL into a single supervised-style training run on preference pairs. Same goal as RLHF, far less machinery. It's the workhorse of open-model alignment because it's stable, cheap, and needs no rollouts or reward model.

Chapter 25

RLVR & reasoning models

The frontier behind reasoning models like o1 and DeepSeek-R1. When correctness is checkable, you don't need a learned reward model at all — you reward with a program. And that changes everything.

Verifiable rewards

For math, code, and many agent tasks, you can check the answer: a math result either matches the solution, code either passes its tests, a task either succeeded. That checking program is the reward function — and it's perfect: ground truth, free to run, and crucially un-hackable (there's no learned proxy to game). This is RLVR: Reinforcement Learning from Verifiable Rewards. The entire reward-hacking problem from Chapter 22 simply vanishes.

GRPO — RL without a critic

GRPO (Group Relative Policy Optimization) deletes the expensive value model from PPO (Chapter 23). For each prompt, sample a group of answers and reward them all. To judge any one answer, compare it to its groupmates: the baseline is simply the group's average reward. Above-average answers get pushed up, below-average down — no value network needed. The advantage is just the reward, standardized within the group:

GRPO advantageAᵢ = ( rᵢ − mean(group) ) / ( std(group) + ε )

Why diversity matters

Notice what the all-correct (or all-wrong) case means: if every sample in a group gets the same reward, all advantages are zero and the model learns nothing from that prompt. GRPO learns only from variation within a group — which is exactly why sampling temperature (Chapter 28) matters during RL: you need diverse attempts, some right and some wrong, to generate a training signal.

The reasoning revolution

The remarkable result: reward a base model only for correct math/code answers (via GRPO + verifiers), and it spontaneously learns to reason — writing long chains of thought, checking its work, backtracking — with nobody showing it how. Just as pretraining forced grammar because it helped prediction, verifiable RL forces reasoning because it helps get answers right. Capability emerges from the objective, one more time. (An honest caveat: pure RL on small models often can't beat distilling from a strong reasoner — the base model matters.)

Simpler cousins are strong baselines: rejection sampling (generate many, keep the verified-correct ones, and just SFT on those) captures much of the benefit with none of the RL complexity.

Chapter 26

RLAIF, Constitutional AI & scalable oversight

Human labels are slow and expensive. Can a model help supervise itself? This is the scalable-oversight agenda — replacing or augmenting human feedback with AI feedback.

RLAIF (RL from AI Feedback) swaps the human labeler for an LLM: given two responses and a rubric, a capable model picks the better one. Same pipeline as RLHF (Chapter 23), but the preferences come from a model — scaling to millions of comparisons cheaply. The risk is inheriting the judge's biases, so rubric design and judge calibration become the quality lever.

Constitutional AI (from Anthropic) makes this concrete with a written set of principles (a "constitution") and runs in two phases. First, a supervised phase where the model improves its own answers through a critique-and-revise loop:

1. draft → model answers a prompt (possibly badly, for a sensitive ask) 2. critique → model is asked to judge its own draft against a sampled principle (e.g. "point out anything that could facilitate harm") 3. revise → model rewrites the answer to fix its own critique 4. train → SFT on the revised answers — and repeat with other principles → no human wrote or labeled any of these responses.

Then a reinforcement phase (this is the RLAIF part): for each prompt the model generates two responses, and the model itself judges which one better follows the constitution; those AI judgments train a reward model, and RL proceeds exactly as in Chapter 23 — just with an AI's preferences instead of a human's. The constitution is the only human input, and it's a short document, not millions of labels.

Intuition — a constitution, a rubric, and a verifier are the same thing

Step back and notice: a constitution (principles), a grading rubric (criteria), and a verifier (a correctness program) are all one object in different clothes — a formalized definition of "good" that a model can apply at scale. The whole game of alignment, across this entire part, is writing down what you want clearly enough that it becomes a training signal. Get the definition right and the optimization mostly follows — which is the second unifying theme of this book.

The broader frontier — debate (models argue opposing sides for a judge), weak-to-strong generalization (can a weaker supervisor elicit a stronger model's full ability?), and LLM-as-judge — tackles the hard question of overseeing systems whose outputs are as hard to evaluate as to produce. These are open problems, but the vocabulary marks the boundary of the field.

Part VI

Inference & Deployment

Training is done; now the model must actually run — generating text, quickly and affordably, at scale. This part covers how generation works and the engineering that makes serving models practical.

Chapter 27

Autoregressive generation

A trained model produces text one token at a time, feeding each output back as input. This simple loop — and its inherent inefficiency — shapes everything about how models are served.

To generate, the model takes the prompt, produces a probability distribution over the next token, picks one, appends it, and runs the whole thing again for the next. This is autoregressive generation, and it splits into two phases with very different performance characteristics:

PREFILL — process the entire prompt in ONE parallel pass (all tokens at once) → fast; produces the first output token. Cost measured as TTFT (time to first token). DECODE — generate the rest one token at a time, each needing a full forward pass → slow & sequential. Cost measured as TPOT (time per output token). a 500-token answer = 1 prefill pass + 499 sequential decode passes
Intuition — why generation is slow

Prefill is fast because the whole prompt is processed in parallel — every token at once. But decoding is fundamentally sequential: you can't generate token 50 until you have token 49, and each new token requires a full forward pass through all the model's layers. This one-at-a-time nature is why long responses take time, why streaming exists (you watch tokens appear as they're decoded), and why nearly all inference optimization targets the decode phase — which is exactly what the next two chapters do.

Chapter 28

Sampling strategies

At each step the model gives a probability distribution; a sampling strategy decides which token to actually emit. This is where "creative" versus "deterministic" behavior lives — and it's all applied after the model, at no training cost.

Temperature — the master dial

Temperature divides the logits by a value T before the softmax. Since softmax exponentiates, dividing by a small T exaggerates the gaps (sharpening toward the top token) while a large T shrinks them (flattening toward uniform). It reshapes the whole distribution without changing the ranking.

Temperature reshapes; then we usually cut off the unreliable tail before sampling, so a rare weird token can't slip through:

Newer variants like min-p (keep tokens above a fraction of the top token's probability) adapt better at high temperature, and a repetition penalty down-weights already-used tokens to avoid loops. Practical guidance: low temperature for code and math (one right answer), higher for brainstorming and creative writing.

Key idea — temperature is also a training tool

Temperature isn't only for inference. It's absent from the standard supervised loss, but it's the exploration engine of RL: rollouts are sampled with temperature (~1.0) to produce the diverse attempts RL learns from — too low and every sample is identical, killing learning (recall GRPO needing group variation, Chapter 25). It also appears inside the distillation loss, softening a teacher's distribution so a student learns its "dark knowledge" — the relative probabilities of the wrong answers. One little division, three different jobs.

Chapter 29

Inference optimization

Serving models cheaply and fast is a huge engineering effort. A handful of techniques do most of the work — starting with the one that makes generation feasible at all.

The KV cache

Naïvely, generating token 500 means re-processing all 499 previous tokens — recomputing their keys and values every single step. But here's the key observation: because of the causal mask, a token's K and V vectors depend only on tokens before it, so they never change once computed. So we cache them. Each new token computes only its own Q/K/V, appends its K/V to the cache, and attends against the stored history. Quadratic waste becomes linear work:

generating 500 tokens WITHOUT a cache (recompute all K/V each step): step 1 processes 1, step 2 processes 2, … step 500 processes 500 total ≈ 1+2+…+500 = 125,250 token-processings (grows with n²) WITH a KV cache (K/V never change once computed): each step processes only the 1 new token → 500 total (grows with n) ⇒ about 99.6% of the work saved

The cache is what makes generation practical — but it isn't free: it's memory that grows with sequence length (and with the number of layers and heads). That memory pressure is exactly why GQA (Chapter 14) exists, letting query heads share K/V to shrink the cache. Three more techniques round out modern serving:

  • PagedAttention (vLLM). Manages the KV cache like an operating system manages RAM — in fixed-size pages — eliminating fragmentation and waste, enabling far more concurrent requests. The basis of modern high-throughput serving.
  • Continuous batching. Instead of waiting for a whole batch to finish, dynamically add and remove requests as they start and complete — keeping the GPU busy and dramatically raising throughput.
  • Speculative decoding. Use a small, fast "draft" model to guess several tokens ahead, then have the big model verify them all in one pass. When the guesses are right (often), you get several tokens for the cost of one — a large speedup with no quality loss, since the big model still approves every token.
Key idea

Inference is dominated by the sequential decode phase and the memory of the KV cache. The wins come from not recomputing what can't change (KV cache), managing that memory cleverly (PagedAttention), keeping the GPU full (continuous batching), and predicting ahead (speculative decoding). Serving frameworks like vLLM and TGI bundle these together.

Chapter 30

Quantization

Models are trained in 16-bit precision, but for inference you can often use far fewer bits per weight — shrinking memory and speeding things up, with little quality loss. This is quantization.

Quantization reduces the numerical precision of the weights: from 16-bit down to 8-bit, 4-bit, or even lower. Because a model's size in memory is just (number of weights × bytes per weight), dropping the bytes per weight shrinks it proportionally — and since inference is often memory-bandwidth bound, a smaller model also runs faster. The savings are dramatic:

a 7-billion-parameter model at different precisions: FP32 (4 bytes/weight) → 7e9 × 4 = 28 GB FP16 / BF16 (2 bytes) → 7e9 × 2 = 14 GB ← the training precision INT8 (1 byte) → 7e9 × 1 = 7 GB INT4 (0.5 byte) → 7e9 × 0.5 = 3.5 GB ← now fits a consumer GPU

The art is doing this without wrecking quality. Naïvely rounding every weight loses too much, so smart methods analyze which weights matter most and preserve their precision while aggressively quantizing the rest:

Intuition — quantization vs QLoRA

Don't confuse the two uses of 4-bit. Inference quantization (GPTQ/AWQ/GGUF) shrinks a finished model so it runs cheaply. QLoRA uses 4-bit as a training trick — a frozen 4-bit base with full-precision adapters on top — so you can fine-tune cheaply. Same idea (fewer bits), different goal (deploy vs train).

Chapter 31

Long context

A model trained on 4,000-token sequences can't natively handle a 100,000-token document. Extending context length — so models can read whole books or codebases — is a distinct challenge with clever solutions.

The problem is twofold. First, attention cost grows with sequence length — quadratically in the naïve form, since every token attends to every other — and the KV cache (Chapter 29) grows linearly on top of that, so memory becomes the binding constraint. FlashAttention (Chapter 14) tackles the compute side. Second, positional encodings trained on short sequences break on long ones: a model simply hasn't seen position 90,000 before.

For that second problem, RoPE (Chapter 7) is the key enabler, precisely because it encodes relative position via rotation — and those rotations can be "stretched." The core trick, position interpolation, is simpler than it sounds: rather than feeding the model position indices it has never seen, you scale them down into the range it was trained on.

a model trained to handle positions 0–4,096, that you want to run at 32,768 (8× longer): naïve → feed positions up to 32,768 → rotation angles never seen in training → breaks position interpolation → divide every position index by 8 before applying RoPE: the token at position 32,000 is rotated as if it were at position 4,000 — in range! → the model reuses angles it already understands; a short fine-tune sharpens the squeezed spacing

YaRN refines this: instead of scaling all the rotation frequencies equally (which slightly blurs the model's sense of nearby tokens), it leaves the high-frequency "local" rotations alone and only stretches the low-frequency "long-range" ones — giving better quality at extreme lengths. This is how the 128K+ context windows you see advertised are actually built: not by retraining from scratch, but by interpolating a short-context model's positions and briefly fine-tuning.

Long context ≠ good long-context use

A model advertising a huge context window doesn't necessarily use it well. A known failure is "lost in the middle" — models attend well to the start and end of a long context but can miss information buried in the middle. Big context is necessary but not sufficient; how reliably the model retrieves from deep in the window is a separate quality question, and a genuine evaluation challenge (Part VII) — which is part of why retrieval (Chapter 36) often beats simply stuffing everything into the prompt.

Part VII

Evaluation

You can build the fanciest pipeline in the world, but if you can't measure whether it improved the model, you're flying blind. Evaluation is the quiet, hard discipline that makes everything else trustworthy.

Chapter 32

Evaluation philosophy

One organizing idea makes sense of every metric: they measure one of two things, and which one matters shifts as a model moves from pretraining to a deployed assistant.

Intrinsic metrics ask "how well does the model model text?" — they look only at the model and some text (loss, perplexity). Extrinsic metrics ask "how good is the model at the actual job?" — they make the model do something and grade the result (accuracy, win rate). Intrinsic is cheap and continuous (a number every training step); extrinsic is what you actually want but slower and harder to measure. The whole difficulty of evaluation is that the cheap thing you can measure constantly is not the thing you actually care about.

Key idea

As a model goes pretraining → fine-tuning → alignment, the useful metrics slide from intrinsic toward extrinsic. Perplexity is your best friend while pretraining and nearly irrelevant after alignment — an aligned model can even have higher perplexity on random web text yet be far more helpful, because it's no longer trying to imitate arbitrary text. "My loss went down" stops being enough; "humans prefer it" takes over.

Chapter 33

The metrics

A tour of what's measured at each stage, from the intrinsic core to the behavioral metrics that define modern chat and reasoning models — with the three most important ones made interactive.

Perplexity (intrinsic)

Evaluation traps

Knowing the metrics is half the job. The other half — the part that separates real practitioners — is knowing how each one lies to you.

  1. Contamination. Test questions leaked into training data (all scraped from the same web), so high scores are memorization, not ability. A 95% benchmark score may be pure leakage. Fix: decontaminate training data against eval sets, and keep private, unseen evals.
  2. LLM-judge bias. When a model grades answers it has predictable biases — position bias (favoring the first shown), length bias (favoring longer), self-preference (favoring its own style). A "win" can be an artifact. Fix: randomize order, swap and average, control for length, spot-check against humans.
  3. Goodhart on the benchmark. Optimize for a benchmark and you improve the score without improving the ability — the model learns the test's quirks. Fix: never optimize against your held-out evals; use a suite, not one number.
The practitioner's instinct

Serious evaluation means a suite of metrics (never one number), decontaminated, unseen evals the model was never tuned against, automatic metrics plus periodic human spot-checks, and deep suspicion of any surprising result. If a recipe improves your dev metrics but not the held-out ones, you optimized the metric — not the model.

Building With LLMs

A trained model is a component, not a product. This part covers the techniques for building useful systems on top of LLMs — from prompting through retrieval, agents, and beyond a single modality.

Chapter 35

Prompting & in-context learning

The simplest and most powerful way to steer a model needs no training at all — just the input you give it. This works because of an emergent property of large models: in-context learning. And in production, prompting alone quietly powers more LLM features than fine-tuning does.

In-context learning is the ability, which emerges at scale (recall Chapter 13), to learn a task from examples given in the prompt, with no weight updates. The prefix conditions the model, and it continues the pattern. That single fact turns a static model into a programmable component.

The prompting toolkit

Prompting is a real engineering skill built from a small set of reliable moves — each one earns its place by fixing a specific failure:

  • Zero-shot — describe the task plainly. Fine when the task is common and unambiguous ("Summarize this in one sentence").
  • Few-shot — show 2–5 input→output examples before the real input. The model infers the pattern, the exact format, and the edge cases from the examples. Reach for it when the output format matters or the task is unusual — examples communicate what instructions can't.
  • Chain-of-thought (CoT) — "think step by step," so the model shows intermediate reasoning before the answer. Buys the model computation (see below); essential for math, logic, and anything multi-step.
  • Self-consistency — sample several independent chains of thought and take the majority answer. Trades cost for accuracy: separate reasoning paths tend to agree on the right answer while disagreeing on the wrong ones.
  • Structured output — demand JSON against a schema. This is what turns the model from a chatbot into a component other code can consume reliably — the backbone of using LLMs in software.
  • Role / system prompt — set persona, tone, and hard constraints once, persistently ("You are a terse senior engineer; never invent APIs").
  • Delimiters — fence the data with markers (triple backticks, XML tags) so the model never confuses your instructions with the user's content — also your first line of defense against prompt injection.

A strong prompt usually stacks these: a role, a clear task, the context/data in delimiters, explicit constraints (length, format, what not to do), and often a few-shot example or two. Order and clarity beat cleverness almost every time.

Q: Roger has 5 tennis balls. He buys 2 cans of 3 balls each. How many now? direct answer → "11" on harder multi-step problems, this blurting often fails chain-of-thought → "Start with 5. 2 cans × 3 = 6 new balls. 5 + 6 = 11." writing the steps gives the model room to compute → reliably right
Intuition — why chain-of-thought works

A model produces each token with a fixed amount of computation. Hard problems need more "thinking" than fits into one token's worth of compute. Chain-of-thought lets the model spread its reasoning across many tokens — writing intermediate steps it can then build on — effectively granting itself more compute to reach the answer. It's the difference between blurting a guess and working it out on paper. Modern reasoning models (Chapter 25) internalize this, "thinking" at length before answering — CoT is the seed of that whole idea.

Real-world use cases

Prompting alone — no fine-tuning, no infrastructure — already runs a huge amount of production LLM work:

  • Classification & routing — tagging support tickets, moderating content, routing emails to the right team, detecting user intent. Often replaces a traditional ML classifier with zero labeled training data.
  • Extraction — pulling structured fields (names, dates, amounts, entities) out of unstructured contracts, resumes, or invoices into JSON your systems can use.
  • Summarization & rewriting — condensing documents, changing tone, translating, simplifying jargon, generating titles.
  • Code — generation, explanation, translating between languages, writing tests, reviewing diffs.
  • Synthetic data — a large model generates labeled examples to train a smaller, cheaper model — a common way to bootstrap a dataset that doesn't exist yet.

Case study — a support-ticket triage classifier

Nothing shows the craft like watching a prompt get iterated against real failures. Goal: label each incoming ticket with a category and an urgency, reliably enough to auto-route it with no human in the loop.

v1 — zero-shot: "Classify this ticket: {text}" → invents categories ("urgent-billing"), inconsistent, sometimes explains instead of labeling v2 — constrain the label set + fence the input: "Categories: billing | technical | account | feedback. Reply with ONLY one category. Ticket: ```{text}```" → categories consistent now, but it confuses ambiguous billing-vs-account tickets v3 — few-shot the hard cases (3–4 examples of the confusing ones) → edge cases fixed; the examples teach the boundary that instructions couldn't v4 — add urgency via CoT + JSON output: "First note any urgency signals, then output {category, urgency: low|med|high}" → urgency is now reasoned, and the output is machine-readable v5 — escalate low confidence: ask for a confidence; if "low", route to a human → the ~5% it's genuinely unsure about no longer get silently mislabeled

The lesson is the method: start simple, look at real failures, and add the one technique that fixes each — constraints, then examples, then reasoning, then structure, then a safety valve. Then measure on a labeled set (Chapter 33). And the exit condition: once you have, say, a thousand labeled tickets and need lower latency and cost per call, you distill this prompt into a fine-tuned small classifier (Chapter 20) — prompting proved the task is solvable and generated the training data.

When prompting isn't enough

Prompting's ceiling tells you which chapter comes next. If the model lacks the knowledge (your private docs, recent events), no prompt fixes that — you need RAG (next). If you need a consistent behavior, format, or style at scale with low latency, bake it in with fine-tuning (Chapter 20). If the task needs actions in the world, you need an agent (Chapter 37). And a caution: any text you drop into a prompt — a retrieved document, a user message, a web page — can contain instructions that hijack the model (prompt injection), which is why delimiters, input sanitization, and never trusting retrieved content as commands are production necessities, not niceties.

Chapter 36

Retrieval-Augmented Generation (RAG)

A model only knows what was in its training data, frozen at a point in time — and it can't cite sources or read your private documents. RAG fixes this by fetching relevant information at query time and giving it to the model. It's the most common way LLMs are put to work over real, proprietary knowledge.

RAG connects an LLM to an external knowledge source. The flow: a question arrives, you retrieve the most relevant passages from a database, insert them into the prompt as context, and let the model generate an answer grounded in them. This gives the model up-to-date, private, or specialized knowledge it never trained on — and lets it cite where each answer came from, sharply reducing hallucination.

The pipeline, stage by stage

Production RAG is much more than "embed and search." Six stages, each with real engineering decisions:

  • Chunking — split documents into passages. The most underrated decision in RAG: too large and retrieval is imprecise and wastes context; too small and each chunk loses the context that makes it meaningful. Practical defaults: ~200–500 tokens with 10–15% overlap so a fact spanning a boundary isn't cut in half, and split on structure (headings, paragraphs, code blocks) rather than blindly by character count.
  • Embedding — turn each chunk into a vector with an embedding model (Chapter 6). The choice matters: domain, language, and dimensionality all affect retrieval quality.
  • Indexing — store the vectors in a vector database with an approximate-nearest-neighbor index (HNSW) so search stays fast at millions or billions of vectors.
  • Retrieval — embed the query and find similar chunks. Best practice is hybrid: combine semantic similarity (meaning) with keyword search (BM25, exact terms) and fuse the rankings — each catches what the other misses.
  • Reranking — over-retrieve (say top-20), then a cross-encoder reranker — which reads query and chunk together rather than comparing pre-computed vectors — re-scores them and keeps the true top-5. Slower per pair, so used only on the shortlist, but a big precision win.
  • Generation — prompt the model with the question and the retrieved chunks, instructing it to answer only from the context and cite which chunk each claim came from.

Real-world use cases

  • Customer support — answer questions grounded in a product's help docs, with citations, so agents (or users) get correct, sourced answers instead of hallucinations.
  • Internal knowledge search — "chat with the company wiki / Slack / Confluence," letting employees ask natural-language questions across scattered internal docs.
  • Legal, medical, financial Q&A — high-stakes domains where every answer must cite its source passage and where the knowledge is private and constantly updated.
  • Code assistants — retrieve relevant files, functions, and docs from a large codebase so the model answers about your code, not generic patterns.
  • Research / analysis — pull the relevant sections from long reports or many papers and synthesize an answer with references.

Case study — why naive RAG fails, and the five fixes

The classic first attempt — split every 1,000 characters, embed, take top-3 by cosine, stuff into the prompt — works in a demo and disappoints in production. Its failures are specific, and each has a known fix:

1. a chunk was split mid-table / mid-sentence → the fact is cut in half fix: structure-aware splitting + 10–15% overlap (e.g. 500-token chunks, 50 overlap) 2. query "how much PTO?" misses a doc that says "vacation allowance" fix: HYBRID search (BM25 keywords + semantic), fused; optionally rewrite the query first 3. the right chunk IS retrieved, but ranked #8 — below the top-5 cutoff fix: over-retrieve top-20, then a cross-encoder RERANKER promotes the true best 4. a chunk says "this limit is 30 days" with no subject — context was lost fix: contextual / parent-document retrieval — prepend the section title or a summary 5. the model has the right context but still hallucinates a number fix: instruct "answer only from the context; cite the chunk; say 'not found' if absent"

There's a cost dividend hiding here, too. Retrieving 5 focused chunks (~2,000 tokens) instead of stuffing all 200 (~80,000 tokens) into the prompt is ~40× cheaper per query — and usually more accurate, because the model isn't distracted by 195 irrelevant passages. Precision beats volume.

Evaluating a RAG system

Because RAG has two stages, a wrong answer has two possible culprits, and you must measure both. Retrieval quality: did the right chunks come back? — measured by recall@k (did we retrieve the relevant chunk at all) and precision@k (how much of what we retrieved was relevant). Generation quality: faithfulness (is every claim in the answer supported by the retrieved context, i.e. no hallucination) and answer relevance (does it actually address the question). Tools like Ragas and LLM-as-judge automate these. The diagnostic discipline: if faithfulness is high but answers are wrong, your retrieval is failing; if retrieval is good but answers drift, your generation prompt is failing. Fix the right half.

RAG vs long context vs fine-tuning

Three ways to give a model what it doesn't have, for different jobs. RAG — large, changing, or private knowledge where you need citations (most enterprise Q&A). Long context (Chapter 31) — when the whole source is small enough to fit and it's a one-off, no retrieval infrastructure needed. Fine-tuning (Chapter 20) — for behavior, format, or style, not facts; it teaches the model how to act, not what is true today. They compose: a fine-tuned model that follows your format, using RAG for the facts, is a common production shape.

Chapter 37

Agents & tool use

On its own, a model can only produce text. An agent wraps the model in a loop that lets it take actions in the world — calling tools, running code, browsing — turning a text predictor into something that gets things done. It's the fastest-moving area in applied LLMs, and the hardest to make reliable.

The core mechanism is tool use (or "function calling"). You describe the available tools to the model (a calculator, a search engine, a database query, an API); when the model decides it needs one, it outputs a structured request to call it; your system runs the tool and feeds the result back; the model continues with that new information. An agent chains this into a loop — think → act → observe → repeat — until the task is done. Step through one:

The anatomy of an agent

Under the loop are a few real components you assemble:

  • Tool schemas — each tool is declared with a name, a plain-language description, and typed parameters. The model reads these descriptions to decide which tool fits and how to fill its arguments, so the wording of a tool's description is real engineering, not documentation.
  • The ReAct pattern — the dominant loop: interleave a Reasoning step ("what do I need next?") with an Acting step (a tool call) and an Observation (the result). It's chain-of-thought (Chapter 35) applied to tool use — the model plans in words before each action, which sharply improves its tool choices.
  • Planning — for complex tasks, the agent first drafts a multi-step plan (subgoals), then executes it, re-planning when reality diverges.
  • Memory — short-term is the running trajectory sitting in the context window; long-term is an external store (often a vector DB — this is RAG, Chapter 36) the agent reads from and writes to across steps or sessions.
  • Reflection — after a failure or a bad result, the agent critiques what went wrong and retries — a self-correction loop that turns one-shot attempts into iterative problem-solving.

Real-world use cases

  • Coding agents (Claude Code, Cursor, and similar) — read files, edit code, run the test suite, and iterate until the tests pass. Tests are a built-in verifier, which is exactly why coding is a sweet spot for agents.
  • Research agents — search the web, read pages, follow leads, and synthesize a cited report — multi-step retrieval no single query could do.
  • Customer-service agents — look up an order, check the refund policy, and actually issue the refund through an API — taking actions, not just answering.
  • Data-analysis agents — write and run code to query a database, plot results, and interpret them, iterating on errors.
  • Browser / computer-use agents — operate software built for humans: navigate UIs, fill forms, click through flows — the frontier of "do this task on my computer."

Case study — the reliability problem, and how to fix it

Here is the single most important fact about agents, and why demos are easy but production is hard: errors compound. Even a strong agent with a high per-step success rate becomes unreliable over a long task, because success multiplies across steps.

task success = (per-step success) ^ (number of steps): 95% per step → over 5 steps: 77% over 10 steps: 60% over 20 steps: 36% ← a "reliable" step, an unreliable task 90% per step → over 20 steps: 12%

That math explains the common failure modes: compounding errors (above); context overflow (a long trajectory fills the window and the agent forgets its earlier steps); loops (repeating a failing action forever); malformed tool calls or misreading a result; and cascading errors (one wrong observation poisons every downstream reasoning step). The fixes are all about fighting the exponent:

  • Shrink the step count — decompose the task and give each subtask a checkpoint, so failures are caught early instead of compounding to the end.
  • Verify each step — check a tool's result before proceeding, and let the agent self-correct on errors (a failing test, a validation error) rather than barreling on.
  • Constrain the action space — fewer, safer, well-validated tools mean fewer ways to go wrong.
  • Human-in-the-loop — require approval before high-stakes or irreversible actions (sending money, deleting data).
  • Budgets & loop detection — cap the number of steps and detect repeated actions so a stuck agent fails fast instead of running up cost.

How do you know if any of this worked? Agent evaluation is end-to-end and task-based: did it complete the task (success rate on a benchmark of real tasks), how many steps and how much did it cost, and did it avoid unsafe actions — a different discipline from the single-output metrics of Chapter 33. And a design rule of thumb: use an agent only when the path can't be pre-scripted. If the steps are fixed and known, a plain chain (a fixed sequence of LLM calls) is cheaper, faster, and far more predictable. Agents buy flexibility at the price of reliability — reach for the simplest thing that solves the task.

Key idea

An agent is a model in a think→act→observe loop with tools, memory, and self-correction. Its power is doing multi-step work no single prompt could; its curse is that per-step errors compound exponentially, so the entire craft is decomposition, verification, constraints, and knowing when a simple chain would do. This closes a loop in the book: training agents in verifiable environments — attempt a task, get graded on success — is the RLVR of Chapter 25, which is why agentic RL is such an active frontier.

Chapter 38

Multimodal models

Language is just one modality. The same transformer machinery extends to images, audio, and video — letting models see, hear, and reason across types of input. And the practical payoff is large: a vision-language model can replace whole brittle pipelines with a single prompt.

The trick is to turn any modality into tokens the transformer can process. A vision-language model (VLM) passes an image through a vision encoder (typically a Vision Transformer, ViT) that splits it into a grid of patches and converts each into a vector, then a small projection layer maps those vectors into the language model's embedding space so they can sit in the same sequence as text tokens. The count is simple geometry:

an image becomes tokens by tiling it into fixed patches: 224×224 image, 14×14 patches → (224/14)² = 256 image tokens 336×336 image, 14×14 patches → (336/14)² = 576 image tokens 512×512 image, 16×16 patches → (512/16)² = 1024 image tokens ≈ a page of text! these image tokens sit in the SAME sequence as text tokens → attention mixes both

The model then attends across image and text tokens together — so it can answer questions about a photo, read a chart, or describe a scene. The architecture barely changes; you add an encoder and a projection that speak the transformer's language of vectors.

How the vision and language halves get connected: CLIP

The encoder has to produce vectors that mean something to the language model — and the technique that makes that possible is contrastive training, of which CLIP is the landmark. Take hundreds of millions of (image, caption) pairs scraped from the web. Encode every image and every caption into vectors. Then train so that a matching image–caption pair has high similarity while every mismatched pairing in the batch has low similarity:

for a batch of N (image, text) pairs, build the N×N similarity grid: the N diagonal entries (correct pairs) → push similarity UP the N²−N off-diagonal entries (wrong pairs) → push similarity DOWN result: a shared space where an image lands near the words that describe it

That shared space is what lets a VLM ground vision in language — and it enables zero-shot recognition: to classify an image, just compare its vector to the vectors of candidate text labels and pick the closest, no task-specific training needed. Audio and video follow the same recipe: an encoder turns a spectrogram or a sequence of frames into vectors in the shared space, and attention does the rest.

Real-world use cases

  • Document understanding — read invoices, forms, receipts, and PDFs without a separate OCR step, understanding layout as well as text.
  • Chart & diagram reading — extract the numbers and trends from a plot or interpret an architecture diagram.
  • Visual Q&A & accessibility — answer questions about an image; describe scenes for blind and low-vision users.
  • Medical & scientific imaging — assist reading X-rays, scans, and microscopy (as decision support, with a human in charge).
  • Robotics / embodied agents — perceive a scene and act on it, connecting vision to the tool-using agents of Chapter 37.
  • Video — summarize footage, find moments, answer questions across time.

Case study — an invoice-extraction pipeline, OCR-free

Goal: pull structured fields (vendor, date, total, line items) out of scanned invoices into JSON. The old pipeline was OCR → regex/templates — brittle, and it broke every time a vendor changed their layout. A VLM collapses it: feed the image directly, prompt for the JSON schema, and the model reads the text and understands the layout in one pass (it knows which number is the total versus a line item because it sees the structure). But naive use fails in specific, instructive ways:

failure → fix, for VLM document extraction: miscounts / drops line items on a long invoice → validate: do line items sum to the total? misreads similar digits (5 vs 6) at low res → raise resolution (more tokens, more cost) hallucinates a field that isn't on the page → allow "not present"; ask it to cite the region tiny fonts / dense tables unreadable → crop & tile the page into higher-res sections

This surfaces the core practical tension of multimodal: resolution versus cost. More pixels means more patches means more tokens (a single 512px image already costs ~1,024 tokens — as much as a page of text), so high-resolution understanding is powerful but expensive, and you tile or downsample deliberately.

Where VLMs still stumble

Vision-language models are strong at description and reading, but reliably weaker at precise counting ("how many people?"), fine spatial reasoning (exact positions, alignment, reading a clock), tiny detail at low resolution, and they will sometimes hallucinate plausible image content that isn't there. For anything high-stakes, verify extracted values programmatically and keep a human in the loop — the same discipline as every other applied chapter in this part.

Intuition — everything becomes tokens

The transformer doesn't care whether a vector came from a word, a patch of an image, or a slice of audio — it just processes sequences of vectors. So "multimodal" is mostly about building encoders that translate each modality into that shared vector language, then letting attention mix them. The same "predict the next token" objective, generalized, drives it — the final echo of this book's first theme: capability emerging from one simple objective, now across sight and sound as well as language.
